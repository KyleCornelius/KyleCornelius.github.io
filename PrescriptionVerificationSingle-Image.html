Prescription Verification Single - Image

 

Introduction

PVS-I (Prescription Verification Single - Image) is a pharmacist application used to verify and fill a single prescription. The most complex medications and highest-risk customers are routed through PVS. Therefore, it was important to be designed in a way that minimizes patient risk, but is also as efficient as possible for our in-house pharmacists.

Goals

  1. Re-use a sister application was previously developed, PVS-E (Prescription Verification Single - Electronic), that will have a lof of the same functionality. Re-use learnings, research, and design components while continuing to re-imagine potential opportunities.
  2. Increase rates at which pharmacists can verify a prescription. Improve the layout to make information easy to consume by reducing clutter/ noise and focusing on the information needed at the time of verification.
  3. Increase System Intelligence Design the system to serve only the data needed to the users. Automate parts of the process that do not require manual intervention.

IRIS (Legacy System)

Image Viewer

For pharmacists to verify imaged prescriptions they must used an application called Image Viewer. Users have dual monitors and have IRIS open on one monitor and the Image Viewer open on the other as they need to continually reference the Image Viewer application for the prescription verification process in IRIS.

Prescription Verification Single - Electronic

With one of the main goals being to re-use components and learnings from PVE-E, a closer look at PVS-E was rquired.

What Worked Well

  • Automated data comparison is efficient and built user trust in the system.
  • Data in one page without scrolling increases efficiency.
  • Colored badges and alerts successfully draw user attention when needed. This successfully decreased error rate
  • Appropriate information appearing on primary view while lower priority information is in a secondary view. This successfully deceased noise and increased efficiency.

Future Opportunities

  • System heavily relies on users to analyze information that is presented
  • System enhancement for the ability to address Prescription holds

PVS-I Considerations

  • 3rd Party Image Viewer will bring challenges to the one page approach taken with PVS-E.

Research for Image Viewer

Most research from PVS-E was reusable but several user interviews, observation sessions, and SME panel sessions were conducted to ensure that assumption was correct and to focus on the key difference, the legacy image viewer.

Key Takeaways:

  • The two screen approach (IRIS on one screen and Image Viewer on another) is very inefficient. Users often lost their spot and their eye-movement range is unreasonably wide.
  • The 3rd party Image Viewer application is very inflexible and clunky. Most of the users time was spent manipulating the images via scroll, drag, or zoom
  • Aside from the image viewer, users behaved like they did for PVS-E

Usability Testing

Multiple prototypes were tested to understand how users interact with them. The overarching goal was to find the strenghts and weaknesses of each prototype and combine that data to iterate upon.

Goals

  • Identify preferred layout through iterative design and usability testing
    • Navigation Errors - excessive mousing and/or scrolling, failure to follow intended screen flow, difficulty finding functionalities
    • Image Viewer Size - excessive zooming, rotating, or general navigation within the image viewer
  • Understand if Action Items:
    • Saves time or is more noise on the screen
    • Contains all the data needed without having to look elsewhere on the application
    • User can easily idenfity unresolved and resolved action items.

A color coded sticky note method was used to categorize user feedback into the following buckets: Likes, Dislikes, Suggestions, and Behaviors. Download PDF version

Findings

  • Users mental model expects the Rx form to the right of the image as that is how it works in PVS-E
  • Horizontal scrolling is a non-starter for most users
  • Users indicated pop out drawers and panels may cause them to miss important data and therefore causes a patient risk
  • Action items panel was received very positively. System calling out what needs to be resolved and being able to resolve it is a very efficient format
  • It is hard to quickly distinguish resolved and unresolved items in action items
  • Users preferred 2 page layout to horizontal scrolling. Ideally want one page without pop out drawers.

Moving Forward

  • Ensure the image viewer is to the left of the Rx Form
  • Remove all pop out drawers/panels
  • Add hot keys with focus on image manipulation as that is where users spend a lot of time
  • Continue with 2 page layout as a one page was deemed not possible without drawers or scrolling
  • Action items resolution needs to be more prominent

Action Items Iterations & Feedback

Parallel to designing the initial wireframes, ideation was happening to introduce more system intelligence. Up to this point, the system presented the information and the application relied on the user to find issues and then use their clinical judgment to resolve issues they found. With the introduction of action items, the system will find and serve up issues to the user with all the information and context needed. The use can then resolve the issue right within the action item form.

This eliminated the need to rely on users to find issues thus reducing errors, rework, and increasing efficiency.

Project Lessons Learned

  • Do not reinvent the wheel. Many of the components were reused from PVS-E, saving a lot of time and user testing.
  • Only deviate from established patterns for a good reason. Mental models are formed with existing patterns. Deviation from those patterns can lead to user confusion.
  • Just because your designs are live in production and has had success, it does not mean it cannot be improved upon. Post production analysis has tangible value.
  • Good design needs lot of iterations
  • Details matter, small changes can have large impact.
